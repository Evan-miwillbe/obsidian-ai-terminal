## MCP & Log Architecture (confirmed 2026-03-31)

Design doc: `docs/MCP연결계획.md`

### MCP Scope
- MCP is **cron schedule CRUD only** (`schedule/set`, `schedule/list`, `schedule/delete`).
- Log writing, hub generation, context delivery are all plugin-internal — NOT MCP tools.

### Log Folder Structure
```
10_Project/{project}/_logs/{host}/{agent}/{YYYY-MM-DD}.md
```
- Each PC writes only to its own `{host}/` folder → no sync conflict.
- `host` is auto-determined via `os.hostname()`.
- Files are append-only, never deleted.
- Summaries go to `_logs/_summaries/_weekly/` and `_monthly/`.

### Hub Notes
- Path: `10_Project/{project}/허브_{project}.md`
- Read-only aggregation view: links, triggers, one-line summaries.
- Generated by cron via `claude -p` (1st), API (2nd), manual (3rd).
- Hub = view, not ledger. Never modify source logs.

### Context Delivery to LLMs
- Symlink hub note into each LLM's context directory.
- Claude Code: `{repo}/.claude/hub-context.md` → symlink to hub note.
- Setup script: `docs/MCP연결계획.md` Part 3 참조.
- No MCP needed — filesystem symlinks are sufficient.

### Scheduler DB
- Shared `schedules.json` between CLI scheduler and MCP schedule tools.
- `ScheduleEntry` extended with optional `action` and `actionInput` fields.
- Deleting a schedule entry does NOT toggle MCP on/off (independent settings toggle).

### Harness vs Log System
- Harness (ruleSync): outbound — vault atoms → external LLM configs.
- Log system: inbound — agent activity → vault logs.
- No overlap. Hub notes must record facts only, never suggest rule changes.

## Named Pipe Architecture (confirmed 2026-04-09)

### Pipe Server: `\\.\pipe\obsidian-ai-terminal`
- JSON-RPC 2.0 over Named Pipe (Win) / Unix Socket (Mac/Linux)
- Started on plugin load, stopped on unload
- Any local process can connect (not just plugin terminals)

### Available Methods
- **Read**: `context/get`, `context/activeNote`, `context/hubs`, `context/recent`, `vault/read`
- **Write**: `vault/write`
- **Control**: `obsidian/openNote`, `obsidian/executeCommand`, `obsidian/listCommands`
- **Terminal**: `terminal/sendKeys`, `terminal/output`
- **ACP**: `agent/list`, `agent/invoke`, `agent/status`, `agent/cancel`
- **Notifications**: `vault/changed` (broadcast to all connected clients)

### wmux Integration (planned)
- Obsidian connects to `\\.\pipe\wmux-daemon` as client for terminal sendKeys
- wmux connects to `\\.\pipe\obsidian-ai-terminal` as client for vault context
- Design doc: `docs/prompt_wmux_named_pipe_integration.md`

### Key Constraint
- `claude -p` headless mode is unreliable (timeout, context limits)
- For real Claude Code session control, use wmux's PTY → sendKeys
- Named Pipe is bidirectional but each side needs its own server
